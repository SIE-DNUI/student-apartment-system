### 2.2 潜在问题发现

#### 问题1：时间段合并逻辑错误（最可能的bug）

**代码位置**：第233行
```python
if merged and start <= merged[-1][1] + timedelta(days=1):
    # 与前一个时间段重叠或连续，合并
    merged[-1] = (merged[-1][0], max(merged[-1][1], end))
```

**问题描述**：
代码将间隔1天的时间段也合并了。例如：
- 时间段A: 1月1日-1月15日 (15天)
- 时间段B: 1月17日-1月31日 (15天)

由于 `1月17日 <= 1月15日+1天(1月16日)`，这两个时间段会被合并为：
- 合并后: 1月1日-1月31日 (31天)

**但实际应该是**: 15 + 15 = 30天

**影响**：这会导致统计的天数比实际多。如果有多个间隔1-2天的换人情况，累计误差会很大。

#### 问题2：归档学生 room_id 可能为 NULL

**代码位置**：第187-193行
```python
students = Student.query.filter(
    Student.department == department,
    or_(
        Student.room_id.isnot(None),
        Student.archived_room_id.isnot(None)
    )
).all()
```

**问题描述**：
查询条件使用了 `or_` 逻辑，但后续处理时：
```python
actual_room_id = student.room_id or student.archived_room_id
```

如果归档学生的 `archived_room_id` 为 NULL，而 `room_id` 也为 NULL，该学生会被查询出来但被跳过：
```python
if not actual_room_id:
    continue
```

**但根据 `fix_archived_rooms.py` 的注释**：
> "由于之前归档时 room_id 被清空，需要手动补充历史房间信息"

这说明历史归档学生的 `archived_room_id` 可能没有被正确设置，导致这些学生的占用时间被忽略。

#### 问题3：同一房间多学生时间段未正确合并

**场景**：
- 学生A在房间101：1月1日-1月15日入住
- 学生B在房间101：1月15日-1月31日入住（1月15日是同一天）

**代码逻辑**：
- 学生A：实际_start = 1月1日，实际_end = min(3月31日, deleted_at或check_out_date)
- 学生B：实际_start = 1月15日，实际_end = ...

**关键问题**：如果两个学生都是1月15日入住/退房，同一天是否应该算作两天？

**当前代码**：
- 学生A：(actual_end - actual_start).days + 1
- 如果退房日=1月15日，入住日=1月1日 → (15-1)+1 = 15天
- 如果新住户1月15日入住 → 同样计算15天

**实际应该是**：
- 学生A: 1月1日-1月14日 (14天)，15日离店
- 学生B: 1月15日-1月31日 (17天)

这样第15天只算一次，但代码会把第15天算两次。

## 三、建议的修复方案

### 修复1：修正时间段合并逻辑

```python
# 修改前（第233行）：
if merged and start <= merged[-1][1] + timedelta(days=1):

# 修改后（只合并重叠的，不合并间隔的）：
if merged and start <= merged[-1][1]:
```

### 修复2：处理同一房间同一天的占用

需要明确定义：
- **入住日（Check-in）**：学生开始使用房间的日期
- **退房日（Check-out）**：学生停止使用房间的日期

如果明确定义"退房日不算"或"入住日算"，需要统一处理。

建议逻辑：
- 如果退房日 == 新住户入住日：退房日不算，新住户从当天开始算
- 代码中 `+1` 是包含首尾两天

### 修复3：确保归档学生房间信息完整

运行 `fix_archived_rooms.py` 脚本检查并修复归档学生的 `archived_room_id`。
